Adversarial behavioral-correctness review of the transport-agnostic session layer and the CAN / UDP platform drivers

📋Executive summary

Cy's consensus core is genuinely strong — the CRDT topic-allocation logic is a faithful, differential-tested implementation of its formal TLA+ model — but the review found one critical memory-safety defect and a cluster of high-severity behavioral bugs at the edges of the design: pattern-subscription discovery, ordered delivery after a peer restart, reliable-delivery acknowledgement semantics, and both platform drivers. Twelve of the findings were reproduced with new tests (Unity intrusive tests and standalone AddressSanitizer probes); the rest are confirmed by direct code tracing.

Overall verdict. The core algorithm is trustworthy and well-modeled; the surrounding machinery is not yet production-hard. The single critical issue (a heap use-after-free reachable in any long-running node that uses reliable delivery or ordered subscriptions) should block release until fixed. Several high issues cause silent data loss or false success reports — the most dangerous class for a reliability-focused library — and several are order- or timing-dependent, so they evade the existing (passing) test suite.

🔬Methodology

The library was reviewed by nine independent max-reasoning agents, each owning a subsystem, plus an orchestrator that read all sources, reproduced findings, and ran an adversarial triage pass that tried to refute the highest-severity claims. Nothing here is accepted on an agent's say-so alone: every High/Critical finding is either backed by a runnable reproduction or by first-hand code tracing recorded below.

Static / model review
  • Full reads of cy.c, cy.h, cy_platform.h, all of model/ (DESIGN, proof, TLA+ Core), and dependency contracts (cavl2, olga_scheduler, wild_key_value, libudpard, libcanard).
  • Line-faithful re-implementations of the CRDT rules in Python, differential-tested against Core.tla.
Empirical validation
  • Debug build of the whole suite: 96/96 tests pass (baseline). Example smoke test over real UDP multicast + SocketCAN vcan + DroneCAN: pass. Rust montecarlo model: 53/53.
  • TLC model-check of CyphalTopics_1 (3 nodes): 5.6M+ distinct states explored with invariants & liveness holding (state space not exhausted at this bound; no violation observed).
  • New file tests/src/test_intrusive_review_repro.c — 8 reproductions; standalone AddressSanitizer probes for the memory-safety and double-free findings.

🛑Critical

C-1
AVL tree is mutated from inside a cavl2_find_or_insert factory → heap use-after-free
criticalreproduced · ASan
Location
cy.c:3064 dedup_factory→dedup_drop_stale cy.c:3398 reordering_cavl_factory→reordering_drop_stale call sites cy.c:3448, cy.c:3497 lib/cavl2.h:444-459
Mechanism

cavl2_find_or_insert walks the tree and captures the parent slot pointer n = &up->lr[x] before invoking the factory (cavl2.h:452-457), then writes the new node through that pointer (*n = out, line 459) and retraces for rebalancing. Both Cy factories call a *_drop_stale() helper that performs cavl2_remove + mem_free on entries of the same tree currently being inserted into. A removal rebalances the tree and can free the captured parent node or leave n pointing at a slot now owned by a live subtree; the subsequent write corrupts the heap, and cavl2_impl_retrace_on_growth then walks freed memory.

Trigger

A topic accumulates several per-remote dedup entries (reliable publishers) or reordering states (ordered subscribers); they go idle for > SESSION_LIFETIME (60 s) while the once-per-poll()-spin round-robin GC has not yet swept that topic; then a reliable message (or ordered message) from a new remote arrives and its insertion fires the in-factory stale purge. This is ordinary behaviour for any long-running node — no adversary, no memory pressure required.

Evidence

Reproduced end-to-end through the real cy_on_message path. AddressSanitizer (NDEBUG, so the library's own balance asserts don't mask it):

==ERROR: AddressSanitizer: SEGV on unknown address ...
  #0 cavl2_impl_rotate           lib/cavl2.h:356
  #3 cavl2_find_or_insert        lib/cavl2.h:464
  #4 on_message                  cy/cy.c:3448
  #5 cy_on_message               cy/cy.c:4822
  #7 test_dedup_factory_mutates_tree_during_insert

tests/src/test_intrusive_review_repro.c :: test_dedup_factory_mutates_tree_during_insert (gated behind -DCY_REVIEW_RUN_UNSAFE; run standalone under ASan)

Fix direction

Hoist the dedup_drop_stale(topic, ts) / reordering_drop_stale(sub, now) calls out of the factories to the call sites, executing them before cavl2_find_or_insert. Optionally document in cavl2 that a factory must not mutate the target tree.

🔴High severity

H-1
Ordered subscription permanently blackholes a remote after it restarts (~50% probability)
highreproduced
Location
cy.c:3256 recency refreshcy.c:3277 backward drop cy.c:3231 reordering_resequencecf. cy.c:3022 dedup reset
Mechanism

The reordering manager linearizes wire tags as lin_tag = tag − tag_baseline, where tag_baseline is frozen at first contact with a remote. When that remote reboots it re-seeds its tag counter with a fresh random 64-bit value, but the local reordering_t for the (remote_id, topic) pair is reused. There is a ~50% chance the new baseline lands in the backward half-space (lin_tag > INT64_MAX, cy.c:3277), in which case every subsequent message is dropped as a late/backward straggler. The only reset path (reordering_resequence) fires only for large forward jumps. Critically, last_active_at is refreshed at the top of reordering_push (cy.c:3256) before the drop check, so each blackholed message keeps the wedged state fresh and the staleness GC never rebuilds it. The blackhole is therefore permanent for any remote republishing faster than 60 s — i.e. essentially all live telemetry. Dedup (cy.c:3022) has a symmetric bidirectional reset; reordering does not — that asymmetry is the bug.

Compounding

For a reliable ordered subscriber this couples with H-3/H-4: the post-restart message is far from the dedup frontier → accepted by dedup → blackholed by reordering → not acked → sender retransmits → now a dedup duplicate → positive ACK sent while the application receives nothing, for every message.

Evidence

test_ordered_subscription_blackholes_after_remote_restart — delivered count frozen after the simulated restart.

Fix direction

Give reordering a bounded straggler window (as dedup has): treat a far-backward lin_tag as a session restart and reordering_resequence, mirroring the large-forward-jump path. At minimum move the recency refresh (cy.c:3256-3257) after the drop checks so the wedged state goes stale within 60 s and is rebuilt.

H-2
Pattern subscribers never bind to topics created locally after the subscription (order-dependent silent loss)
highreproduced · 3 reviewers
Location
cy.c:1652 topic_ensurecy.c:2085 cy_advertise_client cy.c:3643 verbatim subscribeonly router: cy.c:1737 (on_gossip only)
Mechanism

Three code paths create a topic, but only one — topic_subscribe_if_matching, reachable solely from on_gossip for a network-learned unknown topic — routes the new topic against subscribers_by_pattern. A topic created locally by cy_advertise* or a verbatim cy_subscribe is never matched against existing pattern roots, and once it exists by hash, later remote gossips take the known-topic path which never couples. The gap is permanent for the topic's lifetime — no scout or gossip repairs it.

Trigger

cy_subscribe(cy,"a/*") then cy_advertise(cy,"a/b") (or verbatim-subscribe a/b): the pattern subscriber gets no coupling and receives nothing on a/b. In the advertise-only case the topic has no reader at all, so even remote publications never reach the pattern subscriber. Reversing the order works — behaviour is order-dependent, which is why the existing tests (which advertise-before-subscribe) miss it.

Scope

The dominant pattern-subscription use case — discovering topics that remote nodes publish — is unaffected, because those topics are created locally via the gossip path, which does couple. The bug bites specifically when a pattern subscription pre-exists a locally-originated matching topic (this node also advertises or verbatim-subscribes it). Common in nodes that both publish and do discovery; the triage adjudicated it as the high/medium boundary.

Evidence

test_pattern_subscriber_blind_to_later_local_topics — couplings==1 (verbatim only), pattern future receives 0 messages; control (pattern created after the topic) receives them.

Fix direction

After topic_new succeeds in the local-creation paths, route the new topic name through subscribers_by_pattern and couple matches, mirroring topic_subscribe_if_matching (with OOM that reports rather than destroys the topic).

H-3
The documented "fire-and-forget" auto-destroy idiom cancels reliable delivery on the first transient error
highreproduced
Location
cy.h:171 / README.md:159 recommendationexample_file_server.cnotify-while-pending cy.c:2362, 4005
Mechanism

The docs recommend cy_future_callback_set(future, cy_future_destroy) to "destroy itself when done." But the reliable-delivery timeout handlers call future_notify while the future is still pending on every transient send failure (CY_ERR_CAPACITY/CY_ERR_MEDIA) or scheduler-lag detection. With auto-destroy installed, that first transient notification destroys the pending future — which by documented semantics cancels the operation. Retransmissions cease after a single failed attempt, precisely under the congestion that reliable delivery exists to survive. The library's own internal use guards with cy_future_done() first; the public recommendation does not.

Evidence

test_auto_destroy_idiom_cancels_reliable_delivery_on_transient_error — control run retransmits ≥4 times over the budget; with one injected transient failure exactly 2 sends occur, then delivery is abandoned with ~10 s of budget unused.

Fix direction

Provide and document a done-guarded destroy helper (if (cy_future_done(f)) cy_future_destroy(f);) and use it in the README and examples; or separate progress notifications from completion.

H-4
UDP driver: a single zero-length datagram double-frees and crashes every node on the broadcast group
high → borderline criticalreproduced e2e
Location
cy_udp_posix.c:623-627udp_wrapper.c:236-259
Mechanism

A valid 0-byte UDP datagram makes udp_wrapper_receive return 1 with size==0 (recvmsg returns 0, IP_PKTINFO present, iface matches). read_socket then calls realloc(dgram.data, 0), which on glibc frees the buffer and returns NULL (confirmed on glibc 2.35). The guard if (dgram_realloc != NULL) then skips updating dgram.data, leaving it dangling; the freed pointer is handed to udpard_rx_port_push, which fails header validation and frees it again via the deleter — a double-free / heap corruption.

Trigger

Any host sending one empty datagram to the broadcast-subject group 239.127.255.255:9382 — which every Cy/UDP node joins — crashes every receiving node. The design's threat model trusts all nodes but explicitly requires robustness to "network faults and adverse conditions." Triage note: this is remote, unauthenticated, all-nodes heap corruption from a single packet — only the POSIX-transport scope and the glibc realloc(ptr,0)→NULL dependency hold it below critical.

Fix direction

Guard dgram.size==0 before the realloc and free through the memory vtable; also route the shrink through the vtable rather than raw libc so it stays correct under a block-pool allocator.

H-5
CAN driver: best-effort publish on a custom-named pinned topic is silently dropped
highreproduced
Location
cy_can.c:476-497 v_subject_writer_send (13-bit path)cy_can.c:524-532 build_phony_header
Mechanism

For any pinned subject (sid ≤ 8191) with a best-effort header, the writer strips Cy's 24-byte header and transmits on the headerless Cyphal/CAN v1.0 13-bit plane. The receiver cannot recover the topic name, so it fabricates a header with hash = rapidhash(decimal(subject_id)). That only matches a local topic whose resolved name is literally the decimal string of the subject-ID (e.g. "611"). A topic named anything else — including the README's own multi-tenant example foo#1234 / bar#1234 — yields a hash miss and the message is dropped; no 16-bit copy is sent. cy_publish still returns CY_OK.

Evidence

Agent reproduction: control topic 610#610 delivers; foo/bar#611 best-effort is lost while cy_publish_reliable on the same topic (16-bit plane, real header) delivers. This asymmetry — reliable publications, gossips, and N#N-named topics all work — is exactly what makes the best-effort loss look intermittent and easy to miss.

Fix direction

Use the 16-bit plane for pinned v1.1 traffic and gate the headerless 13-bit path behind an explicit v1.0-compat opt-in; or enforce/document that pinned topics must be named N#N.

H-6
CAN driver: responding to an anonymous v1.0 sender indexes a 128-element array at 255 (heap overflow)
highreproduced · ASan
Location
cy_can.c:45 unicast_tid[CANARD_NODE_ID_CAPACITY]=128cy_can.c:715-720 v_unicast_send
Mechanism

Anonymous v1.0 frames arrive with source_node_id = CANARD_NODE_ID_ANONYMOUS = 0xFF (255), stored into lane.id. When the application responds, v_unicast_send indexes owner->unicast_tid[255] — 127 elements past a 128-element array. Debug builds abort on the preceding assert(remote ≤ 127) (DoS); under NDEBUG the out-of-bounds read+write executes before libcanard rejects the ID. (Constants confirmed: array size 128, ANONYMOUS 0xFF.)

Fix direction

In v_unicast_send, early-return CY_ERR_ARGUMENT when remote > CANARD_NODE_ID_MAX instead of asserting/indexing; ideally do not expose a usable lane for anonymous senders.

🟠Medium severity

M-1
Destroying a subscriber future requires an undocumented event-loop spin before cy_destroy
mediumreproduced · 3 reviewers
Location
cy.c:2909-2923 subscriber_disposecy.c:4446 cy_destroy assertcy.c:4139 topic_destroy assert
Mechanism

Unlike publishers (freed synchronously), cy_future_destroy on a subscriber only marks it disposed and arms a 0-deadline timer; the actual root/coupling teardown runs on the next spin. The documented sequence "destroy all user objects, then cy_destroy" (cy.h:541) therefore trips assert(wkv_is_empty(subscribers_by_name)) in debug builds and, in release, destroys topics that still carry live couplings — leaking the subscriber, root, couplings, reordering states, and their message payloads.

Evidence

test_subscriber_destroy_requires_spin_before_cy_destroy — root still registered until a spin runs.

Fix direction

Drain deferred subscriber destruction inside cy_destroy, or document that one cy_spin_once() is required between destroying the last subscriber and cy_destroy.

M-2
A verbatim subscriber's small extent starves a coexisting pattern subscriber's larger extent (silent truncation)
mediumreproduced · 2 reviewers
Location
cy.c:3552-3554 subscription_extent_w_overhead
Mechanism

When a topic has a verbatim coupling, the function executes total = agg; break; — overwriting any accumulated pattern-subscriber extent and skipping the rest — instead of taking the max. With grow-only readers, if the verbatim subscriber couples first (small extent), a later larger pattern subscriber does not grow the reader, so its messages are truncated immediately at steady state. It also makes the effective extent non-monotonic across a subject-ID reallocation (the reader is rebuilt at the recomputed, verbatim-dominated value — a related shrink).

Evidence

test_pattern_extent_starved_by_verbatim — reader extent stays 16+24 while a 4096-byte pattern subscriber is coupled.

Fix direction

Remove the break; take larger() across all couplings unconditionally — extent is a max-message-size guarantee for every subscriber.

M-3
DESIGN.md's modulus-selection rule does not produce the shipped subject-ID moduli (interop hazard)
mediumverified
Location
model/DESIGN.md:46cy_platform.h:20-22Core.tla:33 (stale)
Mechanism

DESIGN.md defines the modulus as "the largest prime ≤ (max subject-ID − 8191) with modulus mod 4 = 3." That rule yields 57331 / 8380403 / 4294959083, but the shipped constants are the smaller 57203 / 8378431 / 4294954663 (deltas 128 / 1972 / 4420). Since a common deterministic mapping is a hard convergence prerequisite (proof assumption A1), a second implementation written from the spec text — rather than by copying cy_platform.h — would compute a different subject-ID for every non-pinned topic and be wire-incompatible. (Note 8380403 is also the stale value in the Core.tla comment.)

Fix direction

Document the actual selection rule (including the gossip-shard reservation that explains the smaller values) in DESIGN.md and refresh the Core.tla comment.

M-4
topic_allocate recursion depth (bounded by topic count) risks stack overflow without tail-call optimization
mediumverified
Location
cy.c:1509 displacement recursioncy.c:1514 self-retry recursioncy.c:1517-1524 CY_TRACE after the call
Mechanism

Each collision-cascade level is one stack frame unless the compiler performs sibling-call optimization, which the standard does not guarantee and which is absent at -O0/-Og and on some embedded compilers. The trailing CY_TRACE unconditionally breaks TCO when tracing is enabled. Depth is bounded by the collision-chain length — at most the local topic count N (not unbounded; termination is guaranteed by CRDT convergence) — but on a many-topic node with dense subject-ID occupancy that can reach N−1 nested frames; a simulation overflowed the interpreter stack at ring 512, and a 2–8 KB embedded stack (~100 B/frame) overflows at a few dozen nested displacements. The code's own comment already flags the TCO dependence and proposes a loop.

Fix direction

Convert the cascade to an explicit loop over a single "current mover" variable (only one topic is in flight at a time).

M-5
CAN driver: single-interface configuration transiently wastes TX-queue capacity on every unicast/service transfer
medium → low-medverified
Location
cy_can.c:721 canard_request (forces CANARD_IFACE_BITMAP_ALL)CANARD_IFACE_COUNT=2
Mechanism

Multicast paths correctly spool on (1<<iface_count)-1, but canard_request/canard_respond spool on all CANARD_IFACE_COUNT compile-time interfaces with no bitmap parameter, while v_spin polls only the real interfaces. Frames destined for phantom interfaces are never drained by a socket, but they are reclaimed by libcanard's tx_expire at their ~1 s deadline — so this is transient TX-queue/memory waste proportional to unicast rate, not a permanent leak (triage correction). It hits every ACK, RPC response, unicast publication, unicast gossip, and scout response whenever iface_count < CANARD_IFACE_COUNT — i.e. the common single-interface SocketCAN setup (cy_can_socketcan_new(1, …)). Existing tests pass because their queues (256) absorb the churn; a small queue could still be exhausted between expiries.

Fix direction

Require iface_count == CANARD_IFACE_COUNT in cy_can_new, or add a bitmap parameter to the libcanard service-transfer calls so they spool only on active interfaces.

M-6
UDP driver: poll() EINTR is reported as a fatal CY_ERR_MEDIA and aborts the event loop
mediumverified
Location
udp_wrapper.c:298-313 (no EINTR retry)cy_udp_posix.c:130-140 err mapping
Mechanism

If a signal interrupts poll(), it returns −1/EINTR; the driver maps any errno except EINVAL/ENOMEM to CY_ERR_MEDIA, which cy_spin_until propagates, ending the spin loop. Any ordinary signal — SIGCHLD from a child process, SIGWINCH on terminal resize, a profiling timer, a debugger attach — stops the node's traffic servicing.

Fix direction

Loop on poll() while errno == EINTR (recomputing the timeout), or map -EINTR to CY_OK so the caller simply re-spins.

M-7
Gossiped topic names are not charset/normalization-validated before creating & re-gossiping implicit topics
mediumverified · 2 reviewers
Location
cy.c:1719 topic_subscribe_if_matchingcy.c:4948-4953 gossip parse
Mechanism

A received gossip name is accepted on length (≤200) and rapidhash(name)==hash only. A malfunctioning remote can gossip names with spaces, control bytes, wildcard tokens, pin-like suffixes, or non-normalized separators; if one matches a local pattern, an implicit topic with that raw name is created, indexed, and re-broadcast once — violating the wire contract "the name is normalized" on the enforcement side. No memory-safety impact (wkv handles arbitrary bytes; the hash check prevents name forgery), but it is a hygiene/interoperability gap that the design's fault-robustness goal calls for closing.

Fix direction

Reject names where name_normalized_len(name) != name.len in topic_subscribe_if_matching.

M-8
A reliable message that no subscriber accepted is still positively acknowledged on retransmit (false delivery report)
mediumreproduced
Location
cy.c:3467 dedup_update (before subscriber loop)cy.c:4838 ack decisionOOM paths cy.c:3324, 3518
Mechanism

on_message records the tag as received in the dedup filter before invoking any subscriber. If acceptance then fails for every subscriber on the first delivery — reordering-slot/state OOM (returns false), or all subscribers disposed-pending-destruction — no ack is sent (correct) and the sender retransmits. On the retransmission dedup_update returns "duplicate" and on_message returns true before touching any subscriber, so a positive ACK is sent for a message the application never received. An in-code comment (cy.c:3462) acknowledges only the narrower "all subscribers disposed" case; the OOM paths are also affected.

Severity note

Triage-adjudicated from high to medium: standalone, the trigger is narrow (OOM, or a subscriber disposed within a single poll cycle). However, it becomes systematic — a false ACK for every message — when compounded with H-1 on a reliable + ordered subscriber whose remote has restarted (dedup accepts, reordering blackholes, retransmit false-ACKs). Treat as high in that combination.

Evidence

test_reliable_false_ack_after_failed_acceptance — first delivery interned, OOM on second, retransmit yields ACK while arrival count stays at 1.

Fix direction

Split dedup_check (consult) from dedup_commit (mark); only commit once at least one subscriber has accepted, so a retransmit after a failed first delivery retries real delivery instead of short-circuiting to an ACK.

✅What is correct — strengths

The CRDT consensus core is a faithful implementation of its formal model. A line-faithful translation of on_gossip_known_topic / on_gossip_unknown_topic / topic_allocate was differential-tested against the TLA+ AcceptGossip / AllocateTopic operators over 5,474,304 exhaustive + 200,000 randomized conflict-free states with zero mismatches, and both subtle empirical-issue fixes documented in the TLA+ README (nonuniform-ageing ping-pong; ignoring collisions from divergent topics unless the local side loses) are correctly implemented. 600+ randomized multi-node networks — including node deaths and unicast-only propagation — all converged.
Verified sound
  • Future framework: "notify is the last access" and "dispose runs only with callback==NULL" invariants hold everywhere, making deep destroy-from-callback chains provably safe.
  • Reliable-delivery backoff schedule and ack_is_last_attempt match the worked example exactly; scheduler-lag detection has zero false positives; the A/B ack-ordering race is genuinely mitigated by the seqno-witness scheme.

🎯Limitations & recommendations

Recommended priority

  1. Fix C-1 (heap UAF) before any release — hoist the stale-sweep out of the cavl factories. It is a latent, time-triggered corruption that will eventually fire in any long-running node using reliable delivery or ordered subscriptions.
  2. Fix the silent-loss cluster (H-1, H-2, M-8). These cause data loss or false delivery success without any error signal — the worst class for a reliability library — and each evades the current suite because it is order- or restart-dependent. M-8 (false ACK) is medium standalone but escalates to high when compounded with H-1.
  3. Fix the driver crashes (H-4 UDP, H-6 CAN) and the CAN pinned-topic loss (H-5); harden the auto-destroy idiom (H-3) in docs and examples.
  4. Reconcile the specification (M-3, L-8, L-9) — the model documents position themselves as the normative basis for independent implementations, so drift here is a real interop hazard.

Scope & limits of this review

  • The association/slack acknowledgement protocol has no formal model backing — only code-level review. The TLA+ models cover CRDT allocation, not reliability. A model of the ack/association machinery would materially raise confidence.
  • CRDT differential testing used the model's small linear-probing ring plus separate quadratic-probing capacity measurements, not full-scale quadratic fuzzing; lock-step timing effects (severe clock skew beyond the proof's stability-window assumption) were analysed, not simulated.
  • Findings marked "verified" are confirmed by first-hand code tracing; those marked "reproduced" have a runnable test. A handful of low/info items rest on library-contract reasoning (cavl2, olga, wkv, libudpard, libcanard) read but not exhaustively fuzzed.
  • This review targets behavioral correctness under the stated trusted-node threat model; it is not a security audit. Several robustness gaps (unbounded state, unvalidated wire names, the zero-length-datagram crash) would nonetheless matter the moment an untrusted participant appears.
